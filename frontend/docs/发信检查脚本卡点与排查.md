# send-flow-check.mjs 卡点清单与排查步骤

本文档按「可能出错的地方」列出卡点，并结合当前实现（**MySQL** + `runSql` / `checkDb`）说明检查与修复方式。脚本入口：`backend/scripts/send-flow-check.mjs`，`--failures` 模式对应 `runFailuresCheck()`。

---

## Ⅰ、入口与参数

### 1. `--failures` 是否被正确识别

**可能问题**：参数分支漏写或顺序导致未进入 `runFailuresCheck()`。

**当前实现**：`main()` 中：

```js
const failuresOnly = args.includes('--failures');
if (failuresOnly) {
  await runFailuresCheck();
  return;
}
```

**检查方式**：在 `main()` 里、`if (failuresOnly)` 分支内第一行加：

```js
console.log('argv:', process.argv.slice(2));
```

执行 `node scripts/send-flow-check.mjs --failures`，应看到 `['--failures']` 且紧接着出现 `========== 失败原因检查 ==========`。

**常见表现**：若没有该标题，说明没进分支（例如拼写成了 `--failure` 或分支被前面的 `--status` 等先匹配）。

**修复**：确认参数为 `--failures`（带 s），且 `if (failuresOnly)` 在 `main()` 中优先于其它模式判断（当前已优先）。

---

### 2. `runFailuresCheck` 是否在调用前已定义

**可能问题**：函数写在文件后面，或导出导致作用域问题，出现 `ReferenceError: runFailuresCheck is not defined`。

**当前实现**：`runFailuresCheck` 与 `runStatusCheck` 均定义在 `main()` 之前，同文件内直接调用，无 `export`。

**检查**：在 `runFailuresCheck` 函数体第一行加 `console.log('runFailuresCheck called');`，执行 `--failures` 时应先看到该行输出。

**修复**：保证函数声明在 `main()` 之上；若拆到其他文件再 `import`，需确认导出/路径正确。

---

## Ⅱ、数据库连接与查询（MySQL）

### 1. `checkDb()` 的返回值用法

**可能问题**：误以为 `checkDb()` 返回了可用的连接/池对象（如 `db`），拿去执行查询，导致 `TypeError: Cannot read property 'query' of undefined`。

**当前实现**：`checkDb()` 只做**连通性检测**，返回 `{ ok: boolean, message: string }`，**不返回**连接或 pool。所有查询都通过 **`runSql(query, params)`** 内部自建 pool 执行。

```js
// runFailuresCheck 中
const dbResult = await checkDb();
if (!dbResult.ok) {
  console.log(`[FAIL] DB: ${dbResult.message}`);
  process.exitCode = 1;
  return;
}
// 后续用 runSql() 查数据，不依赖 dbResult.db
```

**检查**：确认没有出现 `dbResult.db`、`dbResult.pool`、`dbResult.client` 等用法。

**修复**：保持现状：仅用 `dbResult.ok` / `dbResult.message` 判断是否继续，实际查询一律用 `runSql()`。

---

### 2. 环境变量与 DB 连接失败

**可能问题**：`checkDb()` 或 `runSql()` 里用到的 `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME` 未配置或错误，导致连接失败。

**表现**：`[FAIL] DB: ...`，或 `runFailuresCheck` 里 `runSql` 抛错（如 `ECONNREFUSED`、`Access denied`）。

**检查**：在 backend 目录下执行脚本，确保已加载 `backend/.env`（脚本按 `process.cwd()` 找 `.env`，故需 `cd backend` 再执行）。可临时在 `runFailuresCheck` 开头打印：

```js
console.log('DB_HOST:', process.env.DB_HOST, 'DB_NAME:', process.env.DB_NAME);
```

**修复**：在 `backend/.env` 中配置正确的 MySQL 连接信息；在服务器上执行时也要保证 `backend` 为当前目录。

---

### 3. 表名 / 字段名 / 失败状态值

**可能问题**：查询的表或列不存在、失败状态与库里实际存的不一致，导致报错或查不到数据。

**当前实现**：

- 表：`message_tasks`、`accounts`、`account_proxy_bindings`、`proxies`（均为 MySQL 表）。
- 失败状态：`WHERE t.status = 'Failed'`（与 worker 回写一致，首字母大写）。
- 字段：`id, target_phone, status, account_id, error_msg, error_code, content, created_at, updated_at`，以及 JOIN 的账号、代理信息。

**检查**：在 MySQL 里执行等价查询，确认能查到失败记录：

```sql
SELECT id, target_phone, status, account_id, error_msg
FROM message_tasks
WHERE status = 'Failed'
ORDER BY updated_at DESC
LIMIT 5;
```

若库里是 `failed`（小写）或其它枚举，需把脚本中 `WHERE t.status = 'Failed'` 改成实际值（或 `IN ('Failed','failed')`）。

**修复**：表/列与 schema 一致；状态值与 worker 写入的保持一致（当前为 `'Failed'`）。

---

### 4. JOIN 表不存在或缺少列

**可能问题**：`accounts`、`account_proxy_bindings`、`proxies` 任一表不存在，或缺少 `error_msg`、`is_active` 等列，导致 `ER_NO_SUCH_TABLE` / `ER_BAD_FIELD_ERROR`。

**当前实现**：已用 `LEFT JOIN`，缺表时仍可能报错；脚本里对 `ER_NO_SUCH_TABLE` 做了捕获并提示执行 schema。

**检查**：若报错含 `ER_NO_SUCH_TABLE` 或 `ER_BAD_FIELD_ERROR`，在库里执行 `SHOW TABLES;` 和 `DESCRIBE message_tasks;` 等，对照 `backend/scripts/schema_full_create.sql` 或实际建表语句。

**修复**：执行完整 schema，保证上述表和列存在。

---

### 5. `error_msg` 是否被截断

**可能问题**：查询或输出时对 `error_msg` 做了截断，看不到完整失败原因。

**当前实现**：SELECT 中直接查 `t.error_msg`，无 `SUBSTRING`；输出为 `console.log('  失败原因（完整）: ' + (r.error_msg || '(无)'));`，无长度限制。

**检查**：找一条已知较长的失败记录，在库里查 `SELECT LENGTH(error_msg), error_msg FROM message_tasks WHERE id = ?`，对比脚本输出是否一致。

**修复**：保持现状，不对 `error_msg` 做截断。

---

## Ⅲ、脚本逻辑与执行顺序

### 1. `--failures` 与其它参数同时出现

**当前实现**：`failuresOnly` 单独判断且优先，只要 `args.includes('--failures')` 就只执行 `runFailuresCheck()` 并 return，不会再去判断 `--status` / `--create-test`。

**检查**：执行 `node scripts/send-flow-check.mjs --failures --status`，应只做失败检查（因为先命中 `--failures`）。

---

### 2. 运行目录与 `.env` 加载

**可能问题**：在项目根目录执行 `node backend/scripts/send-flow-check.mjs` 时，`process.cwd()` 仍是项目根，脚本从根目录找 `.env`，可能找不到或读到错误配置。

**当前实现**：`loadEnv()` 读取 `path.join(process.cwd(), '.env')`。

**建议**：始终在 backend 目录执行：

```bash
cd /var/www/massmail/backend   # 或本地 backend
node scripts/send-flow-check.mjs --failures
```

这样 `.env` 为 `backend/.env`，与 API/Worker 一致。

## Ⅳ、推荐排查流程（四步）

| 步骤 | 操作 | 期望 | 异常时处理 |
|------|------|------|------------|
| 1 | `node -v` | ≥ 14（建议 18+） | 升级 Node |
| 2 | 在 `if (failuresOnly)` 内首行加 `console.log('argv:', process.argv.slice(2))`，执行 `node scripts/send-flow-check.mjs --failures` | 输出 `['--failures']` 且出现「失败原因检查」标题 | 参数未进入分支 → 检查拼写与分支顺序 |
| 3 | 在 `runFailuresCheck` 首行加 `console.log('runFailuresCheck called');` | 能看到该打印 | 未调用 → 确认上面分支 |
| 4 | 在 `runFailuresCheck` 内、`checkDb()` 后加 `console.log('DB result:', dbResult);` | `dbResult.ok === true` | `ok: false` → 检查 DB 环境变量与 MySQL 是否可达 |
| 5 | 在 MySQL 中执行 `SELECT id, status, error_msg FROM message_tasks WHERE status = 'Failed' LIMIT 3;` | 能返回记录（若业务上确实有失败任务） | 无记录 → 正常，脚本会输出「暂无失败任务」；若表不存在 → 执行 schema |
| 6 | 去掉调试用 console.log，再次执行 `--failures` | 能看到完整 `error_msg` 及代理信息 | 若仍无输出且确认有失败记录 → 核对 `status` 取值与表名 |

## Ⅴ、常见报错与对应原因

| 报错或现象 | 可能原因 | 处理 |
|------------|----------|------|
| 没有「失败原因检查」标题 | `--failures` 未进入分支 | 检查参数拼写、分支顺序 |
| `ReferenceError: runFailuresCheck is not defined` | 函数定义在 main 之后或另文件未正确导入 | 将函数定义移到 main 之前或修正 import |
| `[FAIL] DB: ...` / `ECONNREFUSED` | DB 未启动或 HOST/PORT/凭证错误 | 检查 .env、MySQL 进程、网络 |
| `ER_NO_SUCH_TABLE` | 表未建或库不对 | 执行 schema_full_create.sql 或对应建表 |
| `ER_BAD_FIELD_ERROR` | 列名与 schema 不一致 | 对照实际表结构改 SELECT 或改 schema |
| 输出「暂无失败任务」但库里确有失败 | `status` 与库里不一致（如大小写） | 用 SQL 查实际 status 值，改 WHERE 条件 |
| `error_msg` 显示不完整 | 查询或输出做了截断 | 当前实现无截断；若有自定义修改，去掉 SUBSTRING/切片 |

---

## Ⅵ、与本项目实现对应的要点小结

- **数据库**：MySQL（mysql2/promise），不是 MongoDB；没有 `db.collection()`，只有 `runSql(sql, params)`。
- **checkDb()**：仅返回 `{ ok, message }`，不返回连接对象；后续查询都用 `runSql()`。
- **失败状态**：`message_tasks.status = 'Failed'`（首字母大写）。
- **参数**：`--failures` 已接入且优先；`runFailuresCheck` 在同文件、main 前定义，无需 export 即可被调用。
- **输出**：每条失败任务会打印完整 `error_msg` 及发送时连接 IP（代理），无截断。

按上述卡点逐项检查，即可定位「为什么会这样」的根因。若已出现具体报错或异常输出，把完整报错/终端输出贴出可进一步对症处理。
