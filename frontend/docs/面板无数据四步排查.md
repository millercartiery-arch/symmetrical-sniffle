# 库里有数据但面板「请求任务」为空 — 四步排查

按下面顺序排查，可快速定位“面板没显示”的原因。

---

## 1. 核心原因：多租户隔离（Tenant ID）

**现象**：为支持子账号/操作员引入了 `tenant_id` 过滤，当前登录用户的 `tenant_id` 与库里数据不一致时，接口会只返回该租户的数据，面板看到空列表。

**检查**：在数据库执行：

```sql
USE massmail;
-- 看 message_tasks 的租户分布
SELECT tenant_id, COUNT(*) AS cnt FROM message_tasks GROUP BY tenant_id;
-- 看当前子账号对应的租户（users 表）
SELECT id, username, role, tenant_id FROM users WHERE username = 'subop_20260316_1';
```

若 `message_tasks` 里全是 `tenant_id = 1`，而子账号的 `tenant_id = 2`，则 **GET /api/user/campaigns** 和 **GET /api/user/tasks** 会按租户过滤，返回空。

**解决**：
- 让该子账号的 `tenant_id` 与要查看的数据一致（例如把子账号改为 `tenant_id = 1`），或  
- 在业务允许的前提下，把相关 `message_tasks` / `campaigns` 的 `tenant_id` 改为该子账号的租户。

**代码说明**：`GET /api/user/campaigns` 与 `GET /api/user/tasks` 已按 `req.tenantId`（来自 JWT）过滤，只返回 `tenant_id = req.tenantId` 的数据。

---

## 2. 状态卡死：任务停在 LOCKED 或 Processing

**现象**：Worker 崩溃或 media_url 过长等导致任务未被正常完成，状态一直为 LOCKED/Processing，账号一直 Busy，调度器也不会再把这些任务当“待执行”展示（或展示为进行中但不推进）。

**检查**：执行汇总数量与状态分布：

```sql
USE massmail;
SELECT (SELECT COUNT(*) FROM campaigns) AS campaigns_count,
       (SELECT COUNT(*) FROM message_tasks) AS message_tasks_count,
       (SELECT COUNT(*) FROM accounts) AS accounts_count;
SELECT status, COUNT(*) AS cnt FROM message_tasks GROUP BY status;
SELECT status, COUNT(*) AS cnt FROM accounts GROUP BY status;
```

若 `message_tasks` 有数据但大量处于 LOCKED/Processing，或 `accounts` 大量 Busy，说明需要释放锁。

**解决**：执行「安全释放陈旧锁」脚本，把超时的任务和账号回池：

```bash
# 在服务器上或本机 MySQL 执行
mysql -u 用户 -p massmail < backend/scripts/reset_stuck_tasks_and_locks.sql
```

或打开 `backend/scripts/reset_stuck_tasks_and_locks.sql`，在 MySQL 客户端中执行其中「1. 仅释放超过 5 分钟的陈旧锁」两段 UPDATE。若需强制全部回池（慎用），可使用该脚本中注释的「2. 强制释放所有」两段。

---

## 3. 前端鉴权失败：401 Unauthorized

**现象**：请求 **GET /api/user/tasks** 或 **GET /api/user/campaigns** 时未带有效 token，或 JWT_SECRET 重启后变化，接口返回 401，前端拿不到数据。

**验证**：浏览器 F12 → Network，找到 `user/tasks` 或 `user/campaigns` 请求，看状态码是否为 401 或 403。

**解决**：
- 重新登录，确保请求头带 `Authorization: Bearer <token>`；  
- 确认后端环境变量 `JWT_SECRET` 未在重启后变更（与签发 token 时一致）。

---

## 4. 任务分发逻辑：没有符合条件的“健康账号”

**现象**：调度器只从「健康账号池」取号。若 `accounts` 表里所有账号的 `status` 都不是 `Ready`（例如全是 Cooldown、Dead、Busy），则不会分配任务，任务会一直 Pending，面板上可能只看到“待执行”不减少。

**说明**：代码里健康状态为 **Ready**（不是 ACTIVE）。Busy/Cooldown/Dead 等都不会被选为发送账号。

**检查**：

```sql
SELECT status, COUNT(*) AS cnt FROM accounts GROUP BY status;
```

**解决**：
- 保证至少有部分账号为 `status = 'Ready'`；  
- 若账号因错误被标为 Cooldown/Dead，需按业务逻辑修复或更换账号；  
- 若因之前 Worker 崩溃导致大量 Busy，先执行上面的「释放陈旧锁」脚本，把 Busy 改回 Ready。

---

## 快速确认三件事（可直接发给对接人）

1. **租户匹配**：确认子账号（如 `subop_20260316_1`）的 `tenant_id` 与数据库里 `message_tasks` / `campaigns` 的 `tenant_id` 是否一致。  
2. **脏数据清理**：执行 `backend/scripts/reset_stuck_tasks_and_locks.sql` 中「释放陈旧锁」部分，把卡在 LOCKED/Processing 的任务和 Busy 账号回池。  
3. **API 连通性**：查后端日志，确认 **GET /api/user/tasks** 是否有 401；浏览器 Network 里该请求是否带 Bearer token 且返回 200。

若需「立刻在面板看到任务进入 Processing」，需同时满足：Worker 进程已启动、Redis 正常、有 Ready 账号且带代理、租户一致、无 401。参见 `docs/请求任务不执行排查.md`。
