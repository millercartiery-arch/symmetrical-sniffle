# 修复项与数据库打通检查清单

本文档说明各次修复与数据库的衔接情况，以及如何验证“已打通”。

---

## 1. Dashboard（修复文件：看板统计）

| 项目 | 说明 |
|------|------|
| **前端** | `useDashboardStats` 请求 `GET /api/dashboard/stats`；`StatsPanel` / `GlobalSignalPanel` 展示数据。 |
| **后端** | `backend/src/routes/dashboard.ts` → `router.get("/dashboard/stats", ...)`，使用 `pool.getConnection()` 查询数据库。 |
| **涉及表** | `message_tasks`（任务数、今日发送/失败）、`accounts`（总数、Ready/Cooldown/Dead）。 |
| **连接** | 后端从 `../shared/db.js` 引入 `pool`（MySQL 连接池），与 `.env` 中 `DB_*` 一致。 |

**验证方式：**

```bash
# 1) 后端能连库且表存在
mysql -u massmail -p massmail -e "SELECT COUNT(*) FROM message_tasks; SELECT COUNT(*) FROM accounts;"

# 2) 接口返回正确结构
curl -s -H "Authorization: Bearer <JWT>" http://localhost:3000/api/dashboard/stats | jq .
# 期望：{ "task": { "total", "inProgress", "completed", "failed" }, "account": { "total", "online", "offline", "cooldown", "todaySent", "todayFailed" } }
```

## 2. Proxy 外键与 Schema（修复文件2）

| 项目 | 说明 |
|------|------|
| **前端** | 无直接调用；代理相关 API 由账号/代理池等页面使用。 |
| **后端** | `ensureProxySchema(conn)` 在以下时机执行：<br>• 任意代理路由请求（`proxy.ts` 中间件）<br>• 卡密/凭证 schema 初始化（`card-credential-schema.ts`）<br>• 调度器每轮（`scheduler.ts`） |
| **涉及表** | `proxies`（id 为 BIGINT UNSIGNED）、`proxy_audit`（proxy_id BIGINT UNSIGNED）、`account_proxy_bindings`、`sub_accounts.proxy_id`。 |
| **写入** | 代理增删改测时，`proxy.ts` 向 `proxy_audit` 插入 `INSERT INTO proxy_audit (proxy_id, action, operator_id, detail)`。 |

**验证方式：**

```bash
# 1) 表结构与外键
mysql massmail -e "SHOW CREATE TABLE proxies\G"
mysql massmail -e "SHOW CREATE TABLE proxy_audit\G"
# 期望：proxies.id 与 proxy_audit.proxy_id 均为 BIGINT UNSIGNED

# 2) 已有库需先跑迁移（若之前为 INT）
mysql massmail < backend/scripts/migrate_proxy_audit_fk.sql

# 3) 代理操作后应有审计记录
mysql massmail -e "SELECT id, proxy_id, action, operator_id, created_at FROM proxy_audit ORDER BY id DESC LIMIT 5;"
```

## 3. 隐私弹窗（PrivacyModal + 审计落库）

| 项目 | 说明 |
|------|------|
| **前端** | `PrivacyModal` 在用户选择后调用 `api.post('/privacy/status', { status: 'accepted' | 'rejected' })`；部分页面可能调用 `POST /api/audit/privacy`（body.action）。 |
| **后端** | • `POST /api/privacy/status`：将用户选择写入 `audit_logs`（action=`privacy_consent`，details=`{ status }`）。<br>• `POST /api/audit/privacy`：将审计动作写入 `audit_logs`（action=`privacy_audit`，details=`{ mode }`）。 |
| **涉及表** | `audit_logs`（user_id, action, details, tenant_id）。 |

**验证方式：**

```bash
# 1) 表存在（与 schema_full_create.sql 一致）
mysql massmail -e "SHOW CREATE TABLE audit_logs\G"

# 2) 前端点击「同意并继续」或「不出售我的信息」后，应有记录
mysql massmail -e "SELECT id, user_id, action, details, created_at FROM audit_logs WHERE action IN ('privacy_consent','privacy_audit') ORDER BY id DESC LIMIT 5;"
# details 为 JSON，如 {"status":"accepted"} 或 {"mode":"..."}
```

## 4. 数据库连接与表依赖总览

| 修复项 | 使用的连接 | 读表 | 写表 |
|--------|------------|------|------|
| Dashboard 统计 | `pool`（dashboard.ts） | message_tasks, accounts | — |
| Proxy Schema | `pool`（proxy-schema.ts），由 proxy/account/scheduler 传入 conn | — | 建表/ALTER + proxy_audit 由 proxy 路由写入 |
| 隐私选择 | `pool`（dashboard.ts） | — | audit_logs |

**环境变量（后端）：** `DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`（默认 `massmail`）。  
**前端 API 基址：** `VITE_API_BASE_URL`（如 `https://your-domain/api`），请求会发到该基址下的 `/dashboard/stats`、`/privacy/status` 等。

## 5. 一键检查（可选脚本）

在项目根目录或 backend 下执行（需已配置 MySQL 客户端与 .env）：

```bash
# 表是否存在
mysql -N -e "SELECT TABLE_NAME FROM information_schema.TABLES WHERE TABLE_SCHEMA=DATABASE();" massmail | grep -E 'message_tasks|accounts|proxies|proxy_audit|audit_logs'

# 最近审计
mysql massmail -e "SELECT action, details, created_at FROM audit_logs ORDER BY id DESC LIMIT 3;"
```

若上述表均存在，且接口返回 200 与预期 JSON，则修复项已与数据库打通。
