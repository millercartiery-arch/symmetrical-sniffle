# 导入账号之后的使用逻辑

本文说明本项目中「账号从哪来」以及「导入/写入账号后如何被调度与发送」的完整链路，与当前实现（TextNow + Puppeteer、BullMQ、`account_proxy_bindings`）对齐。

---

## 1️⃣ 本项目的「后台导入」是怎么来的？

| 方式 | 是否需要前端 UI | 说明 |
|------|----------------|------|
| **CLI / 脚本 / 直接写库** | **不需要** | 在服务器或本机执行脚本、或通过内部 API/运维工具把账号写入 MySQL 的 `accounts` 表即可。业务只依赖表中已有数据。 |
| **原 UI 导入** | 已下线 | 原 `POST /api/tn-accounts/import` 与账号管理页「Import」已移除，接口现返回 404。批量导入请使用脚本或内部工具。 |

**为何用脚本而不是只在页面上点「导入」？**

- **安全**：账号含密码、session、代理等敏感信息，公开上传入口易被滥用。
- **批量效率**：千、万条用 CSV/JSON + 批量 `INSERT` 比网页逐条提交快得多。
- **运维友好**：可在 CI、部署脚本、ansible 中直接调用，无需额外前端。

若业务需要在管理后台提供导入，应做成**受鉴权的内部 API**，后端复用同一套「解析 CSV/JSON → 写 `accounts`」逻辑，而不是重新实现。

## 2️⃣ 导入完毕以后，账号「被谁、怎么使用」？

### 2.1 业务模型（关键表）

| 表名 | 关键字段 | 作用 |
|------|----------|------|
| **`accounts`** | `id`, `status`, `tenant_id`, `locked_at`, `locked_by`, 以及登录/设备相关字段 | 所有可发信账号。`status='Ready'` 且未被锁超时 → 可被调度器选用。代理来自 **`account_proxy_bindings`** + `proxies`，不是 `accounts.proxy_url` 单列。 |
| **`message_tasks`** | `id`, `campaign_id`, `target_phone`, `content`, `status`, `account_id`, `tenant_id`, `scheduled_at`, `retry_at`, `locked_at`, `error_msg` | 发送任务的原子单元：Pending → LOCKED → Processing → Sent / Failed。`account_id` 可为空（未分配）或已指定。 |
| **队列（逻辑）** | 无物理 `task_queue` 表 | 队列 = 查询 `message_tasks` 中 `status IN ('Pending','PENDING')` 且 `scheduled_at <= NOW()` 且 `(retry_at IS NULL OR retry_at <= NOW())` 的记录，由调度器用 **`FOR UPDATE SKIP LOCKED`** 抢占。 |

### 2.2 调度器（Scheduler）如何「挑选」账号？

调度逻辑在 **`backend/src/workers/scheduler.ts`** 的 `dispatchPending()` 中：

1. **取待发任务**：从 `message_tasks` 中取满足上述条件的任务（带 `FOR UPDATE SKIP LOCKED`），并 JOIN `campaigns` 得到 `tn_account_ids`（活动可选账号范围）。
2. **决定使用的账号**：
   - **若任务的 `account_id` 已有值**（例如创建活动时指定了「使用账号」）：校验该账号 `status='Ready'`、属当前租户、且在活动的 `tn_account_ids` 内（若有）；不满足则本轮跳过该任务。
   - **若 `account_id` 为空**：从 `accounts` 中选一条满足 `status='Ready'`、`tenant_id` 一致、未锁超时、且（若有 `tn_account_ids`）`id IN (tn_account_ids)` 的账号，按 **`last_used_at ASC`** 选最少使用的。
3. **代理校验**：通过 `account_proxy_bindings` 解析该账号的代理 URL；**无绑定代理则不派发**该任务（本轮跳过）。
4. **占锁与派发**：Redis 分布式锁占住 `accountId`，把任务更新为 `LOCKED`、`account_id`、`locked_at`，把账号更新为 `Busy`、`locked_at`、`locked_by`，然后将任务推入 BullMQ 队列 `tn-send`。
5. **Worker 消费**：Worker 从队列取任务，用该账号 + 绑定代理通过 Puppeteer/TextNow 发信，成功后任务置为 `Sent`、账号置回 `Ready` 并释放 Redis 锁；失败则按错误类型回滚（见《TN账号列队执行注意事项》）。

**要点**：

- 导入的账号只要在 `accounts` 中且 **status='Ready'**、**有代理绑定**、**租户正确**，就会自动进入「可选账号池」。
- 任务创建时可以不填 `account_id`，由调度器按策略自动分配；也可以填（活动指定账号或轮询预填），调度器会校验后使用。

## 3️⃣ 在 UI/后台「指定」使用哪个账号？

| 场景 | 操作方式 | 在本项目中的对应实现 |
|------|----------|------------------------|
| **不指定账号** | 只填目标号码、内容等，任务的 `account_id` 留空 | 创建活动时不选「使用账号」或 `tn_account_ids` 为空 → 插入的 `message_tasks.account_id` 为 null → 调度器按 Ready + 租户 + 代理 + 可选 `tn_account_ids` 自动挑一个。 |
| **指定一批账号（活动级）** | 在创建活动时选择「使用账号」列表 | `campaigns.tn_account_ids` 存所选账号 ID 数组；插入任务时可按目标列表**轮询预填** `account_id`（见下），调度器只会在这些 ID 中选用或校验。 |
| **指定单个账号** | 单条任务绑定某账号 | 插入 `message_tasks` 时直接写 `account_id = ?`；调度器会校验该账号 Ready 且属当前租户（及在活动允许列表内），通过则用该账号派发。 |

**创建活动时的预填逻辑**（`backend/src/routes/campaign.ts`）：

- 若创建活动时传了 `tn_account_ids`（前端选的账号列表），插入 `message_tasks` 时会按目标列表 **轮询分配** `account_id`：  
  `accountId = allowedIds[index % allowedIds.length]`，这样「使用账号」列可立即显示，且调度器会优先/仅使用这些账号。

## 4️⃣ 从「导入」到「使用」的完整闭环

```
  ┌─────────────────┐       ┌─────────────────────┐       ┌─────────────────────┐
  │ 1) 账号来源      │       │ 2) 任务创建         │       │ 3) 调度与发送        │
  │ CSV/脚本/内部API │  -->  │ message_tasks       │  -->  │ Scheduler → Worker  │
  │ 写入 accounts    │       │ (Pending, account_id│       │ 占锁 → BullMQ → 发信 │
  └─────────────────┘       │  可为空或已指定)     │       └─────────────────────┘
         ^                   └─────────────────────┘                 ^
         │                              ^                             │
         │                              │ 活动/聊天/入站 创建任务        │ 状态回写
         │                              │                             │
         └──────────────────────────────┴─────────────────────────────┘
                    账号池 (Ready + 代理) 被调度器选用
```

1. **导入/写入账号**：通过 CLI、脚本或内部 API 写入 `accounts` 表；在「账号管理」中完成登录探测、在「IP 配置」中绑定代理后，账号变为 **Ready + 有代理**，进入可用池。
2. **创建任务**：活动创建、聊天发信、入站回复等会向 `message_tasks` 插入记录（status=`Pending`），可带或不带 `account_id`、可受 `campaigns.tn_account_ids` 限制。
3. **调度**：Scheduler 定时取 Pending 任务，为无 `account_id` 的任务选账号（或校验已有 `account_id`），解析代理，占 Redis 锁与 DB 锁后推入 `tn-send` 队列。
4. **发送**：Worker 消费队列，用账号 + 代理执行 TextNow 发信，写回 Sent/Failed 与 `error_msg`，释放账号锁与 Busy 状态。
5. **排查**：`node scripts/send-flow-check.mjs --failures` 查看最近失败任务的完整错误与代理 IP。

## 5️⃣ 常见问题 FAQ（本项目）

| Q | A |
|---|---|
| 导入后账号是否立刻可被 Worker 使用？ | 是。写入 `accounts` 后，只要 **status 为 Ready**、**已绑定代理**（`account_proxy_bindings`）、**租户一致**，下次调度轮询就会进入候选池。若导入时 status 为其它值，需在「账号管理」中改为 Ready 或重新探测。 |
| 任务创建时要不要填 account_id？ | 不必。不填则调度器按 Ready + 租户 + 代理 + 活动可选账号自动分配。若需要「只用某几个账号」，在创建活动时选择「使用账号」即可（写入 `tn_account_ids` 并可能预填 `account_id`）。 |
| 指定了 account_id 是否一定立刻发送？ | 不一定。调度器会校验该账号是否 Ready、是否有代理；若该账号正被占用（Redis 锁或 status=Busy），本任务会留在 Pending，下一轮再试。 |
| 有多少 Worker 合适？ | 受 Redis、MySQL 连接数和「单账号同时只发一条」限制。账号数较多时 2～4 个 Worker 通常足够；可配合监控 pending 堆积与 Busy 数量调整。 |
| 导入失败（如某行缺字段）怎么处理？ | 若使用脚本导入，建议脚本**按行收集错误**，整批跑完后输出 inserted/updated/failed 统计，不因单行错误中止，便于修正脏数据。 |
| 想快速验证账号是否可用？ | 可创建一条测试任务发到自己的号码，或使用 `node scripts/send-flow-check.mjs --create-test <手机号>` 触发一次发送；结合 `--failures` 查看是否出现 Login failed、代理错误等。 |

## 6️⃣ 推荐运维 SOP（本项目）

1. **准备账号**：CSV/脚本或内部工具写入 `accounts`；在「账号管理」完成登录/探测，在「IP 配置」绑定代理，确认 status 为 Ready。
2. **检查账号池**（可选）：  
   `SELECT status, COUNT(*) FROM accounts WHERE tenant_id = ? GROUP BY status;`  
   确认 Ready 数量符合预期。
3. **启动/扩容**：确保 Scheduler 与 Worker 进程在跑（如 pm2），按需扩容 Worker。
4. **创建任务**：通过活动、聊天或 API 插入 `message_tasks`（可不填或按需填 `account_id`）。
5. **监控**：pending/sent/failed 数量、账号 Busy/Cooldown、队列堆积（见《TN账号列队执行注意事项》Ⅵ）。
6. **故障排查**：定期执行 `node scripts/send-flow-check.mjs --failures`，根据 `error_msg` 处理 Login failed、代理错误、限流等。
7. **定期清理**：对长期 LOCKED/Busy 的僵尸锁，调度器会按超时自动释放；必要时可执行文档中的释放 SQL。可定期清理过旧的 Failed 任务（按业务保留策略）。
