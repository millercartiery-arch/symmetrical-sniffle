# 域名访问白屏排查

访问 https://hkd.llc 或 https://你的域名 出现白屏时，按下面顺序检查。

## 1. 服务器是否做了 SPA 回退（最常见）

前端是单页应用（React Router），**所有路径**（如 `/`、`/login`、`/admin/dashboard`）都必须返回同一个 `index.html`，由前端路由接管。否则直接访问 `/login` 会 404，表现为白屏。

**Nginx 示例**（站点根目录即前端静态文件时）：

```nginx
server {
    listen 443 ssl;
    server_name hkd.llc;
    root /var/www/massmail/frontend/dist;   # 前端构建产物目录
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;   # 关键：找不到文件时回退到 index.html
    }

    location /api {
        proxy_pass http://127.0.0.1:3000;    # 后端 API
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /socket.io {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

若前端不在根路径（例如在 `/app/`），需在 `frontend/vite.config.ts` 中设置 `base: '/app/'`，并重新构建；Nginx 的 `root` 和 `try_files` 也要对应到该目录。

## 2. 浏览器控制台是否有报错

- 打开 https://你的域名，按 **F12** 打开开发者工具，切到 **Console**。
- 若有红色报错（如 Failed to load script、CORS、Uncaught TypeError），根据报错修：
  - **脚本 404**：静态资源路径或服务器未正确部署 `dist` 下的文件。
  - **CORS**：后端需允许该域名（`ALLOWED_ORIGINS` 含该域名），或 Nginx 代理到同源。
  - **其他 JS 报错**：可能是环境变量或接口地址不对，见下一条。

## 3. 生产构建时的环境变量

前端生产包在**构建时**会把 `frontend/.env.production` 里的变量打进代码。确认：

- `VITE_API_BASE_URL=https://hkd.llc/api`（或你的实际 API 地址）
- 若 API 和页面同域名，可写相对路径 `/api`，由 Nginx 代理到后端。

修改 `.env.production` 后需重新构建并部署：

```bash
cd frontend && npm run build
# 再把 dist 目录部署到服务器
```

## 4. 静态资源是否 404

在 F12 的 **Network** 里刷新页面，看是否有请求 404（尤其是 `/assets/xxx.js`、`/assets/xxx.css`）。若 404：

- 确认 Nginx `root` 指向的目录里确实有这些文件（即 `dist` 完整上传）。
- 若站点不在根路径，检查 `vite.config.ts` 的 `base` 与 Nginx 配置是否一致。

## 5. 后端与接口是否正常

- 在浏览器或本机执行：`curl -I https://hkd.llc/api/health`（或你的健康检查接口），确认返回 200。
- 若 API 不通，登录会失败；若前端在请求接口时抛错且未捕获，也可能导致白屏。结合第 2 步 Console 报错排查。

## 6. 首次进入是旧版页面，刷新后变成新版（图1 → 图2）

这是**浏览器或 CDN 缓存了旧的 index.html** 导致的。入口 HTML 被缓存后，会继续加载旧的 JS，所以看到旧版登录样式（如「欢迎登录」「数字验证」）；刷新后拿到新 index.html，就变成当前版本（「登录」「验证码」「换一张」）。

**处理方式：**

- **Nginx**：对 `index.html` 单独加不缓存头，例如在 `server` 里增加：
  ```nginx
  location = /index.html {
      root /var/www/massmail/frontend/dist;   # 你的前端根目录
      add_header Cache-Control "no-cache, no-store, must-revalidate";
  }
  ```
  然后 `nginx -t && nginx -s reload`。这样每次打开或刷新都会向服务器要最新的 index.html，从而加载最新前端资源。
- **用户侧**：可先强制刷新（Ctrl+F5 或 Cmd+Shift+R）清掉本地缓存，之后就会稳定显示新版。

**总结**：域名白屏多数是**未配置 SPA 回退**（`try_files $uri $uri/ /index.html`）。先加上该配置并重载 Nginx，再结合 F12 Console/Network 排查其余问题。若「首次旧版、刷新新版」则按上面为 index.html 设置不缓存。
