# 发信打通测试

按下面顺序在**服务器**上执行，可快速确认发信链路是否通畅。

## 零、查看发送情况（检查某号码或最近任务）

只查数据库里任务状态，不检查 Redis/账号：

```bash
cd /var/www/massmail/backend
node scripts/send-flow-check.mjs --status
```

只看发往某号码（如 18706956266）的任务：

```bash
node scripts/send-flow-check.mjs --status 18706956266
```

会列出该任务的 id、状态（待处理/已分配/发送中/已发送/失败）、账号、时间、错误信息。

**专门查看失败原因**（最近 20 条失败任务，完整错误信息 + 账号信息）：

```bash
node scripts/send-flow-check.mjs --failures
```

会输出每条失败任务的：目标号码、使用的账号(id/username/状态)、**发送时连接 IP（代理）**、失败原因（完整 error_msg）、账号侧错误（若有）。末尾会提示常见原因（登录失败→改密码/探测；代理超时→检查 IP 配置）。

---

## 发信是否带代理 IP（连接 IP）

**是。** 每条发送任务都会通过**该账号绑定的代理 IP** 发出：

1. 调度器派发任务时，会从「账号 → account_proxy_bindings → proxies」解析出代理（或账号表上的 `proxy_url`），没有代理则**不派发**该任务。
2. Worker 收到任务后，用该代理启动 Puppeteer（`--proxy-server` + 认证），TextNow 发信流量走该代理出口。
3. 运行 `node scripts/send-flow-check.mjs --failures` 时，会打印每条失败任务对应的**发送时连接 IP（代理）**，便于排查是否代理不可用导致失败。

---

## 一、运行发信打通检查脚本（推荐先做）

在服务器上进入 backend 目录，执行：

```bash
cd /var/www/massmail/backend
node scripts/send-flow-check.mjs
```

脚本会检查：

- **Redis** 是否可连接（调度器与 Worker 依赖 Redis 队列与锁）
- **数据库** 是否可连接
- **Ready 账号数**、**有代理绑定的账号数**、**Ready 且带代理（可发信）** 数量
- **当前 Pending 任务数**
- **最近 3 条失败任务** 的 `error_msg`

若输出中有 `[FAIL]` 或提示「没有 Ready 且带代理的账号」，按提示先修再测。

---

## 二、可选：插入一条测试任务并观察是否被发送

在**确保 Worker 已启动**的前提下，可插入一条测试任务并轮询 90 秒看是否变为 Sent。**建议使用您自己的手机号**作为目标号码，便于在手机上确认是否收到测试短信。

```bash
cd /var/www/massmail/backend
node scripts/send-flow-check.mjs --create-test 您的手机号
```

例如（美国 10 位号码，可带国家码 1）：

```bash
node scripts/send-flow-check.mjs --create-test 8706956266
# 或
node scripts/send-flow-check.mjs --create-test 18706956266
```

- 脚本会选一个 **Ready + 有代理绑定** 的账号，插入一条 Pending 任务。
- 若 90 秒内任务变为 **Sent**，说明发信链路已打通。
- 若一直 Pending 或变为 **Failed**，请结合「三、四」查 Worker 与日志。

---

## 三、确认两个进程都在跑

```bash
pm2 list
```

应看到：

- `massmail-api`：online（提供 API）
- `massmail-worker`：online（调度 Pending 任务 + 消费队列实际发信）

若没有 `massmail-worker`：

```bash
cd /var/www/massmail/backend
pm2 start dist/worker-entry.js --name massmail-worker
pm2 save
```

---

## 四、看 Worker 日志（任务不执行或失败时必看）

```bash
pm2 logs massmail-worker --lines 100
```

关注：

- `No available accounts`：没有 Ready 账号或租户/活动账号过滤后为空。
- `No proxy binding for account=xxx`：该账号未绑定代理，需在「IP 配置」里绑定。
- `Missing proxyUrl`、`Account lock not acquired`：代理或锁异常。
- `Task xxx failed`、`TextNow automation failed`、`Login failed`：登录或发送失败，看 `error_msg` 与日志细节。

---

## 五、常见阻塞点对照

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| 检查脚本报 Redis FAIL | Redis 未启动或 REDIS_URL 错误 | 启动 Redis，检查 backend/.env 中 REDIS_* |
| Ready 且带代理 = 0 | 账号不是 Ready，或未绑定代理 | 账号管理里恢复 Ready；IP 配置里为账号绑定代理 |
| 有 Pending 任务但不执行 | Worker 未起、Redis 异常、或时间熵排到未来 | 起 Worker、查 Redis；可查 message_tasks.scheduled_at |
| 任务变为 Failed | TextNow 登录失败、代理不可用、页面超时 | 看 error_msg 与 pm2 logs，修账号/代理或重试 |
