# 数据‑信号显示面板 验收手册

目标：确认 **前端面板** 能完整、准确、实时地展示后端的信号（proxy / 子账号 / 卡密）数据，且 **前后端通路** 已全部连通、授权、可靠。

---

## 1️⃣ 功能 / 交互验收（手动 UI 检查）

| 步骤 | 操作 | 预期表现 |
|------|------|----------|
| 1.1 登录 & 进入面板 | 管理员登录 → 进入「IP 配置」Tab | 页面打开，无 401/404，URL 正确，请求带 `Authorization: Bearer <jwt>`。 |
| 1.2 列表渲染 | 首次加载 | 表格分页（默认 1–20 条），列：协议、地址、描述、标签、权重、状态 Switch、地区、延迟、上次检测、操作。`enabled` 与 DB 一致。 |
| 1.3 过滤 / 搜索 | 输入关键字（如主机、地区）→ 搜索 | 请求带 `search=…`（及可选 `protocol`、`region`、`enabled`），表格仅匹配行。 |
| 1.4 分页 | 切到第 2 页 | 请求带 `page=2&pageSize=20`，表格与后端一致，显示「共 X 条」。 |
| 1.5 新增代理 | 新增 → 填写表单 → 保存 | 新行出现，`bind_count=0`，DB 中 `auth_pass_enc` 有值、无明文密码返回前端。 |
| 1.6 编辑代理 | 编辑 → 改 weight/description → 保存 | 行更新，`proxy_audit` 有 `action=update`。 |
| 1.7 启用/禁用 | 切换行内 Switch | 立即 PATCH，DB `is_active` 更新。 |
| 1.8 删除代理 | 删除 → 确认 | 软删（`is_active=0`），子账号 `proxy_id` 自动置空，表格刷新，`proxy_audit` 记 `action=delete`。 |
| 1.9 连通性测试 | 点击「测试」 | 弹出成功/失败与延迟；表格更新 `last_success_at` / `last_latency_ms` / `last_error_msg`。 |
| 1.10 绑定子账号 | 账号详情 → 绑定代理 → 保存 | `sub_accounts.proxy_id` 更新，调度使用该代理。 |

---

## 2️⃣ 数据完整性 & 端到端通路

| 检查点 | 预期结果 |
|--------|----------|
| **GET /api/proxies** | 返回 `list` + `total` + `pagination`，每条含 protocol, host, port, description, tags, weight, enabled, region, last_* , **bind_count**，**不包含** password/auth_pass_enc。 |
| **bind_count** | = account_proxy_bindings 数 + sub_accounts.proxy_id 数；GET /api/proxies/:id 与列表一致。 |
| **错误结构** | 缺必填返回 400，body `{ error: "..." }`；唯一键冲突返回 **409**，前端提示「同协议、主机、端口已存在…」。 |
| **POST /api/proxies/:id/test** | 返回 `{ ok, latency, status }`，DB 更新 last_success_at / last_latency_ms / last_error_msg。 |
| **CORS** | 跨域请求 Response Headers 含 `Access-Control-Allow-Origin`，无 CORS 报错。 |
| **审计** | CRUD + Test 均写 `proxy_audit`，含 `proxy_id`, `action`, `operator_id`, `detail`。 |
| **软删 + 解绑** | DELETE 后 `proxies.is_active=0`，且 `sub_accounts.proxy_id` 置 NULL（业务层更新）。 |

---

## 3️⃣ 自动化测试 & CI

- **单元/服务层**：`tests/proxy*.test.ts` — 密码加密、test 成功/失败、bind_count、软删、批量事务回滚、审计。
- **运行**：`npm run test:proxy`（若已配置）、`npm test`。
- **E2E**（可选）：Playwright/Cypress 登录 → 新增 → 绑定 → 测试 → 删除 全流程。
- **Husky pre-commit**：`npm run lint && npm run type-check && npm test`；可选 `git-secrets --scan` 防明文泄露。

---

## 4️⃣ 监控 / 回滚 / 故障排除

| 监控项 | 正常阈值 | 故障排查 |
|--------|----------|----------|
| 代理连通性 | 成功率 ≥ 95% | 查 `proxy_audit` 最近 `action=test` 的 error。 |
| bind_count | 与 account + sub_account 表一致 | 对比 COUNT(*) 与 API 返回。 |
| API 响应时间 | 95% ≤ 200 ms | 查 DB 索引、EXPLAIN。 |
| 软删后残留 | `sub_accounts.proxy_id` 指向已禁用的 proxy 可视为需清理 | 业务层软删时已置空，无需额外清理。 |

**回滚**：密码泄露 → revert 提交、恢复/轮换密钥；误删 → `UPDATE proxies SET is_active=1 WHERE id=?`；整体 → `pm2 stop all` + 回退代码再部署。

## 附：常用调试命令

```bash
# bind_count 一致性
mysql -e "
SELECT p.id, p.host, p.port,
       (SELECT COUNT(*) FROM account_proxy_bindings b WHERE b.proxy_id = p.id AND b.is_active=1) AS acct_cnt,
       (SELECT COUNT(*) FROM sub_accounts s WHERE s.proxy_id = p.id) AS sub_cnt
FROM proxies p;
"

# 最近 5 条审计
mysql -e "SELECT * FROM proxy_audit ORDER BY created_at DESC LIMIT 5\G"
```
