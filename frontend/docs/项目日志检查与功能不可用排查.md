# 项目日志检查与「很多功能无法使用」排查

## 一、当前日志来源

- **请求审计日志**：`backend/logs/requests.audit.log`  
  每行一条 JSON：时间、方法、路径、状态码、耗时、IP、userId、tenantId、UA。

---

## 二、从日志里看到的主要问题

### 1. 大量 401 Unauthorized

| 路径 | 含义 |
|------|------|
| `POST /api/login`、`POST /api/auth/login` | 登录失败：用户名/密码错误，或后端未正确配置管理员密码 |
| `POST /login` | **路径错误**：后端只有 `/api/login`，没有 `/login`，说明前端请求的 base URL 可能少了 `/api` |
| `POST /api/tn-accounts/import` | **已下线**，现返回 404；若日志仍出现 401 多为旧版前端或书签在调用 |
| `POST /user/campaigns` | 未带 token 或 token 无效/过期，导致需要登录的功能不可用 |

### 2. 路径不一致

- 日志里出现 **POST /login**、**POST /api/auth/login**。  
  后端实际提供的登录接口是：**POST /api/login** 和 **POST /api/auth/login**（两个都有）。  
  若前端或第三方只请求 **/login**（没有 `/api` 前缀），会 404 或由反向代理返回 401。

### 3. 登录成功后的接口也 401

- 例如 **POST /user/campaigns** 返回 401，说明：  
  - 要么请求没带 `Authorization: Bearer <token>`  
  - 要么 token 过期/无效（JWT 过期时间 15 分钟，需注意刷新或重新登录）

---

## 三、为什么「很多功能无法使用」——对应关系

1. **登录就失败**  
   → 无法拿到 token → 所有需要登录的接口都会 401，表现为「功能都不能用」。

2. **前端请求的 API 地址不对**  
   → 例如 base 配成 `http://服务器:3000` 而不是 `http://服务器:3000/api`，  
   → 会请求到 `/login`、`/user/campaigns` 等，与后端路由不一致，404/401。

3. **后端环境变量未配置**  
   → 生产环境必须配置 `ADMIN_PASSWORD_HASH`（及可选的 `OPERATOR_PASSWORD_HASH`）、`JWT_SECRET`。  
   → 未配置或配错会导致登录永远失败或 token 验证失败。

4. **数据库缺少 users 表**  
   → 若未执行 `schema_full_create.sql`，没有 `users` 表，仅靠环境变量的 admin 账号可以登录，但若环境变量也没配好，就会全部 401。

---

## 四、建议逐项检查（按顺序做）

### 1. 后端环境变量（服务器或本地 .env）

在 **backend** 目录下检查：

```bash
# 必须存在且不能是占位符
ADMIN_USERNAME=admin
ADMIN_PASSWORD_HASH=$argon2id$...   # 用 argon2 生成的真实哈希，不要用 REPLACE_WITH_ARGON2_HASH
JWT_SECRET=至少32字符的随机字符串
```

生成 Argon2 哈希示例（Node）：

```bash
cd backend && node -e "const argon2=require('argon2'); argon2.hash('你的密码').then(h=>console.log(h))"
```

生产环境若未配置 `ADMIN_PASSWORD_HASH`，登录会直接失败。

### 2. 前端 API 基础地址

- 开发/测试：`frontend/.env.development` 或 `.env.production` 中  
  `VITE_API_BASE_URL` 应指向**当前使用的后端**，且**带 `/api` 后缀**。  
  例如：`http://localhost:3000/api` 或 `https://你的域名/api`。  
- 若配成 `http://服务器:3000`（没有 `/api`），前端会请求 `/login`、`/api/me` 等，  
  后端实际是 `/api/login`、`/api/me`，路径对不上会导致 404/401。

### 3. 数据库与 users 表

- 确认已执行 `backend/scripts/schema_full_create.sql`（或同等建表脚本）。  
- 若使用数据库里的用户登录，需在 `users` 表中有对应用户，且密码为 Argon2 哈希。

### 4. 登录后仍 401（例如 /user/campaigns）

- 确认请求头带：`Authorization: Bearer <登录返回的 token>`。  
- 若长时间未操作，token 可能过期（15 分钟），需重新登录或实现 refresh。

---

## 五、如何自己再看一遍日志

- 看最近错误：  
  `backend/logs/requests.audit.log` 最后几行，关注 `status` 非 200 的条目。  
- 按路径统计（PowerShell，在项目根目录）：  
  `Get-Content backend\logs\requests.audit.log | ForEach-Object { ($_ | ConvertFrom-Json).path } | Group-Object | Sort-Object Count -Descending`  
  可看到哪些路径被调得多、方便对照 401 是登录还是业务接口。
- 或使用脚本快速看摘要与错误：  
  `node backend/scripts/analyze-audit-log.cjs`

---

## 六、小结

| 现象 | 优先检查 |
|------|----------|
| 登录一直失败 | 后端 `ADMIN_PASSWORD_HASH`、`JWT_SECRET`，以及 `users` 表是否存在 |
| 请求路径是 /login 不是 /api/login | 前端 `VITE_API_BASE_URL` 是否以 `/api` 结尾、是否指对后端地址 |
| 登录成功但其他功能仍 401 | 请求是否带 Bearer token、token 是否过期、后端 JWT_SECRET 是否一致 |

按上面逐项检查并修正后，再抓一次 `requests.audit.log` 看 401 是否减少、功能是否恢复。
