# 测试发送未到达（8706956266 等）— 排查步骤

## 一、先确认任务在系统里的状态

在**服务器**上执行，看发往 8706956266 的任务当前状态和报错信息：

```bash
mysql -h 127.0.0.1 -u massmail_app -pry2nwSP4nbcBDSL3 massmail -e "
SELECT id, target_phone, status, account_id, error_msg, created_at, updated_at
FROM message_tasks
WHERE target_phone = '8706956266'
ORDER BY id DESC
LIMIT 5;
"
```

- **status = Pending**：任务还没被 Worker 执行（调度器/Worker 没跑、或 Redis/账号/代理有问题）。
- **status = Failed**：执行过但失败，看 **error_msg** 列的具体报错。
- **status = Sent**：系统侧已标记「已发送」，若手机仍没收到，多半是通道/号码/运营商问题（见下文）。

---

## 二、看 Worker 日志（失败时必看）

在服务器上：

```bash
pm2 logs massmail-worker --lines 100
```

关注是否有：

- `Task xxx failed: ...`（会把错误写进 message_tasks.error_msg）
- `No proxy binding`、`Account lock not acquired`、`Missing proxyUrl`
- Puppeteer/TextNow 相关超时、登录失败、元素找不到等

根据报错修代理、账号或环境。

---

## 三、常见原因对照

| 现象 | 可能原因 | 建议 |
|------|----------|------|
| 任务一直是 Pending | Worker 未启动、Redis 挂了、无 Ready 账号、账号无代理 | 见《请求任务不执行排查》：起 Worker、查 Redis、保证有 Ready+代理 |
| 任务变成 Failed，有 error_msg | 代理不可用、TextNow 登录失败、页面超时、被限流 | 看 error_msg 和 pm2 logs，修代理或换账号 |
| 任务已是 Sent 但手机没收到 | TextNow 仅支持 App/Web 内会话、号码格式不对、运营商/地区限制 | 确认 8706956266 在 TextNow 里能收到；号码带国家码如 +18706956266 视实现是否支持 |

---

## 四、号码格式

- 库里 **target_phone** 一般是 10 位（如 8706956266），发到 TextNow 时是否需要加国家码（如 1）取决于后端/TextNow 的拼接逻辑。
- 若接口或 Worker 要求「带国家码」，可尝试把目标写成 **18706956266** 或 **+18706956266** 再测一次。

---

## 五、建议操作顺序

1. 在服务器执行上面的 **SQL**，确认 8706956266 对应任务的状态和 **error_msg**。
2. 若 status = Failed，执行 **pm2 logs massmail-worker**，把相关错误贴出来再排查。
3. 若 status = Pending，按《请求任务不执行排查》确保 Worker 在跑、Redis 正常、有 Ready 且带代理的账号。
4. 若 status = Sent 但手机未收到，检查号码格式（是否需国家码）以及 TextNow 是否对该号码/地区有送达限制。

把 SQL 查到的 **status** 和 **error_msg**（以及必要时 logs 里一两行关键报错）发出来，可以进一步精确定位。
