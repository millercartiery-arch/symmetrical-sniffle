### 3. 分布式环境的负载均衡与时域调度（Temporal Distribution Entropy）✅ 已落实

| 子项 | 状态 | 代码位置 |
|------|------|----------|
| 行为建模算法（泊松分布/时隙） | ✅ 已实现 | `backend/src/workers/scheduler.ts`：`samplePoissonIntervalSeconds`（指数分布近似泊松）、`temporalCursorByCampaignMs` 按 campaign 的时域游标、`scheduled_at` 写入 DB。 |
| 分布式协调 / Redis 锁 | ✅ 已实现 | `scheduler.ts`：`acquireAccountRedisLock` / `releaseAccountRedisLock`；`worker.ts` 侧也有 account lock。 |
| 住宅 IP 不硬编码 / 节点资源 | ✅ 已实现 | `proxies` + `account_proxy_bindings` 表；scheduler 中 `resolveProxyUrlForAccount` 从绑定表取代理并带 Session；Worker 仅用 job 内 `proxyUrl`。见 `PROXY_MODULE_CHECK.md`、`CLOSING_CHECKLIST.md`。 |

**为何可能「看不到」：**  
- 时域调度：需要有待发任务（Pending）、且 campaign 配置了 `min_interval`/`max_interval`，调度器才会给任务打上 `scheduled_at` 并延后执行；若没有任务或间隔为 0，界面上看不到明显「错峰」效果。  
- 住宅 IP：需在「IP 池管理」中先添加代理、再为账号绑定，列表和状态才会在接口/前端显示；若未配置绑定，发信仍可能走 `accounts.proxy_url` 或直接跳过。

---

### 4. 无状态会话的容错与热迁移架构（Session High-Availability）✅ 部分落实

| 子项 | 状态 | 代码位置 |
|------|------|----------|
| locked_at 状态与超时释放 | ✅ 已实现 | `scheduler.ts`：`releaseStaleLocks` 按超时释放 task/account 的 `locked_at`；schema 含 `locked_at`、`scheduled_at`。 |
| TCP RST / 链路异常识别 | ✅ 已实现 | `worker.ts`：`classifyFailure` 将 ECONNRESET、socket hang up 等归为 `TCP_RST`；失败时 `releaseAndRollback` 将任务回退为 Pending、账号 Ready/Cooldown。 |
| 影子环境热切换 | ✅ 已实现 | `backend/src/gateway/dispatch-engine.ts`：`SHADOW_FAILOVER_ENABLED`、主 session 失败后尝试 shadow session、`recordShadowFailoverEvent`；通过 `SHADOW_FAILOVER_MAX_ATTEMPTS` 等环境变量可配。 |

**为何可能「看不到」：**  
- 影子切换在 **协议网关（DispatchEngine）** 路径生效，用于通过 gateway 发信的会话；若你当前主要用 **scheduler → worker → Puppeteer/TextNow** 这条线，则不会经过 DispatchEngine，界面上不会看到「影子切换」的日志或状态。  
- TCP RST 处理体现在：任务失败后变为「可重试」、账号进入 Cooldown；需在任务列表/账号状态中查看，而不是单独「链路监测」面板。

## 三、总结

- **优化方案第 3、4 条**（时域调度、住宅 IP 管理、locked_at 释放、TCP RST 处理、影子容错）已在代码中落实，对应文件见上表。  
- **第 1、2 条**（TLS/JA3/JA4、HTTP/2 流控）未在本仓库实现，因此界面上不会有相关「修复」或开关。  
- **「很多修复没有看到」** 的常见原因：  
  1. 代理/绑定为空，未在 IP 池管理中添加并绑定账号；  
  2. 没有 Pending 任务或 campaign 未设间隔，时域调度无表现；  
  3. 主要走 scheduler→worker 线路而非 DispatchEngine，因此看不到「影子切换」；  
  4. 项目未做 git init，所以没有「提交」记录，但磁盘上的修改已存在。  

建议：在「IP 池管理」中配置代理与绑定后，创建一条带间隔的 campaign 并产生任务，再观察任务列表与账号状态；若需版本可追溯，再按上节执行 `git init` 与首次提交。
