# 全局打通验收手册（0→1 全链路）

在 Staging/Production 服务器上一键验证：**环境 → 迁移 → 种子数据 → 调度+Worker → 任务成功/失败**。

---

## Ⅰ 前置条件

| 条目 | 说明 |
|------|------|
| Node | 20+ |
| MySQL | 已部署，库 `massmail` |
| Redis | `REDIS_URL` 或 REDIS_HOST/PORT，供调度与 Worker |
| .env | backend 目录下，含 DB_*、PROXY_ENC_KEY（可选）、JWT_SECRET、REDIS_* |
| pm2（可选） | 进程守护：`npm i -g pm2` |

**.env 示例：**

```bash
DB_HOST=127.0.0.1
DB_PORT=3306
DB_USER=massmail
DB_PASSWORD=xxxx
DB_NAME=massmail
PROXY_ENC_KEY=$(openssl rand -base64 32)
REDIS_URL=redis://127.0.0.1:6379
JWT_SECRET=$(openssl rand -hex 32)
```

---

## Ⅱ 数据迁移与表结构

**首次部署（从零建库）：**

```bash
mysql -u root -p -e "CREATE DATABASE IF NOT EXISTS massmail;"
mysql -u root -p massmail < backend/scripts/schema_full_create.sql
```

**已有库（仅补表/列）：**

```bash
cd backend
npm run db:migrate
# 或
npm run db:ensure
```

---

## Ⅲ 种子数据（credential / card_key / sub_account / proxy）

```bash
cd backend
npm run seed
```

输出四个 ID：credentialId、cardKeyId、subAccountId、proxyId。子账号已绑定该 proxy，供调度器使用。

环境变量可覆盖默认值：`SEED_CRED_USERNAME`、`SEED_CRED_PASSWORD`、`SEED_CARD_CODE`、`SEED_PROXY_HOST`、`SEED_PROXY_PORT`、`SEED_PROXY_USER`、`SEED_PROXY_PASS`。

---

## Ⅳ 创建测试任务与启动 Worker

**1. 启动 Scheduler + Worker（单进程）：**

```bash
cd backend
npm run start:worker
# 或使用 pm2
pm2 start npx --name massmail-worker -- tsx src/worker-entry.ts
```

**2. 插入测试任务（由调度器自动分配子账号或账号）：**

```bash
node scripts/send-flow-check.mjs --create-test 13800138000
```

- 若有 **Ready + 代理** 的 account，任务会预填 `account_id`。
- 若无 account 但有 **ready + enabled + proxy_id** 的 sub_account，则插入 `account_id=NULL`，调度器会自动分配 sub_account。

**3. 查看结果：**

```bash
node scripts/send-flow-check.mjs --failures
node scripts/send-flow-check.mjs --status
```

---

## Ⅴ 一键全局验证脚本

在 **backend** 目录下：

```bash
chmod +x scripts/full-run-check.sh
./scripts/full-run-check.sh
```

脚本将依次：加载 .env → npm ci → db:ensure → seed → 启动 pm2 worker → 执行 --create-test → --failures → 打印最近 proxy_audit。

## Ⅵ 代理健康探测与密钥轮换

**代理探测（更新 last_checked_at / last_alive / last_error_msg）：**

```bash
node scripts/proxy-check.mjs
```

或通过 API：`GET /api/system/proxies/status`（需后端已启动）。

**代理密码密钥轮换（更换 PROXY_ENC_KEY 后重加密）：**

```bash
npm run migrate:proxy-pass   # 明文 → auth_pass_enc 迁移
# 若仅轮换密钥（解密后再用新密钥加密）：
tsx scripts/rotate-proxy-keys.ts
```

## Ⅶ 验收清单

| 步骤 | 命令/操作 | 预期 |
|------|------------|------|
| 1 | `npm ci` | 依赖安装完成 |
| 2 | `npm run db:migrate` 或执行 schema_full_create.sql | 表/索引/外键存在 |
| 3 | `npm run seed` | 输出 4 个 ID |
| 4 | 启动 Redis，`npm run start:worker` 或 pm2 启动 worker-entry | 无报错 |
| 5 | `node scripts/send-flow-check.mjs --create-test 13800138000` | 任务插入，轮询后显示 Sent/Success 或 Failed+原因 |
| 6 | `node scripts/send-flow-check.mjs --failures` | 若有失败，显示完整 error_msg |
| 7 | `node scripts/proxy-check.mjs` | 启用代理的 last_* 更新 |
| 8 | 查库：`SELECT * FROM proxy_audit ORDER BY created_at DESC LIMIT 5` | 有 create/test 等记录 |
| 9 | 前端「IP 配置」查看代理、bind_count、状态 | 与 DB 一致 |

## Ⅷ 故障排查

| 现象 | 可能原因 | 建议 |
|------|----------|------|
| 任务一直 Pending | 无可用子账号/账号（ready、enabled、有 proxy） | 查 `sub_accounts` / `accounts` 与 proxy 绑定 |
| Worker 报 ECONNREFUSED | 代理不可达或认证错误 | 用 proxy-check.mjs 或 API 测试代理；检查 auth_pass_enc |
| 无 Ready 账号 | 未跑 seed 或 account 未绑定 proxy | 跑 seed；或为 account 绑定 proxy（account_proxy_bindings） |
| 子账号未分配 | 调度器未选到 sub_account（无 proxy_id、非 ready） | 确认 sub_accounts.proxy_id 已设，status=ready、enabled=1 |

详细说明见项目内「数据‑信号显示面板」与「IP 配置面板」验收文档。
