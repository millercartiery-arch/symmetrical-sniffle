# 请求任务没有执行 — 排查与修复

## 一、执行链路（谁在干活）

请求任务（`message_tasks` 里 status=Pending）要真正发出去，需要两条腿同时跑：

| 环节 | 所在进程 | 作用 |
|------|----------|------|
| **调度器** | Worker 进程 | 每 2 秒从数据库捞 Pending 任务，分配账号、代理，推到 Redis 队列 `tn-send` |
| **Worker** | 同一 Worker 进程 | 消费 `tn-send` 队列，用 Puppeteer/TextNow 实际发消息 |

这两个都在 **同一个入口** 里：`backend/src/worker-entry.ts`（编译后 `dist/worker-entry.js`）。  
**只起 API（index.js）不会起 Worker**，所以不会有人去捞任务、也不会有人消费队列，任务就会一直 Pending。

---

## 二、最常见原因：没起 Worker 进程

部署脚本当前只执行了：

- `pm2 start dist/index.js --name massmail-api`

没有执行：

- `pm2 start dist/worker-entry.js --name massmail-worker`

所以服务器上只有 API，没有「调度器 + 队列消费者」，请求任务不会执行。

### 修复：在服务器上起 Worker

SSH 上服务器后，在项目 backend 目录执行：

```bash
cd /var/www/massmail/backend
pm2 start dist/worker-entry.js --name massmail-worker
pm2 save
```

以后每次部署完，也要保证 worker 在跑，例如：

```bash
pm2 restart massmail-api massmail-worker
# 或
pm2 restart all
```

（若已在 deploy 里加上对 massmail-worker 的 start/restart，则部署后会自动带上。）

---

## 三、其他会导致任务不执行的原因

在 **已经启动了 Worker 进程** 的前提下，若任务仍不执行，可对照下面检查。

### 1. Redis 未运行或连不上

调度器和 Worker 都用 Redis（队列 `tn-send`、分布式锁）。Redis 挂了或连不上，调度器会跳过本次调度，Worker 也收不到任务。

- 检查：`redis-cli ping` 应返回 `PONG`。
- 环境变量：`REDIS_HOST`、`REDIS_PORT`、`REDIS_PASSWORD`（或 `REDIS_URL`）需与当前 Redis 一致。

### 2. 没有可用账号（Ready）

调度器只会把任务分给 `accounts` 里 **status = 'Ready'** 的账号（且满足租户、可选 `tn_account_ids` 限制）。若没有 Ready 账号，日志里会出现类似：

- `No available accounts for task xxx, tenant=1`

处理：在「账号管理」里确认有账号，且状态为 Ready；若被标成 Cooldown/Dead，需按业务逻辑恢复或更换账号。

### 3. 账号没有代理（Proxy）

调度器在分配账号后会查该账号的代理（表 `account_proxy_bindings` 或 `accounts.proxy_url`）。若没有配置代理，该任务会被跳过，日志类似：

- `No proxy binding for account=xxx, skip task=xxx to keep session consistency`

处理：给该账号配置代理（绑定或 `proxy_url`），或确认业务上是否允许无代理（若代码强制要求代理则必须配置）。

### 4. 任务被「时间熵」延后

若开启了 `ENABLE_TEMPORAL_ENTROPY`（默认开），任务会先被赋予 `scheduled_at`，到点才会被捞出来执行。若 `campaign` 的 `min_interval`/`max_interval` 设得很大，任务会看起来「一直没执行」，其实是还没到调度时间。

处理：查库确认 `message_tasks.scheduled_at` 和当前时间；或临时关掉熵：`ENABLE_TEMPORAL_ENTROPY=false`。

---

## 四、如何确认 Worker 在跑

- 本机：看进程里是否有 `node dist/worker-entry.js` 或 `tsx src/worker-entry.ts`。
- 服务器：`pm2 list` 里应有 `massmail-worker`，且 status 为 online；`pm2 logs massmail-worker` 能看到调度/发送相关日志（如 "Found N pending tasks"、"Dispatched task x to account y"）。

---

## 五、小结

| 现象 | 优先检查 |
|------|----------|
| 任务一直是 Pending，从没变过 | 是否启动了 **massmail-worker**（`dist/worker-entry.js`） |
| Worker 已起仍不执行 | **Redis** 是否正常；是否有 **Ready** 账号；账号是否配置 **代理**；是否被 **scheduled_at** 推到未来 |

先保证「API + Worker 两个进程都跑、Redis 正常、有 Ready 且带代理的账号」，请求任务才会被调度并真正执行。

## 六、群发历史里「状态」列说明（为什么一直显示待处理）

状态来自数据库 `message_tasks.status`，前端对应关系如下：

| 后端 status | 前端显示 | 含义 |
|-------------|----------|------|
| Pending / PENDING | 待处理 | 任务在队列中，尚未被调度器分配账号 |
| LOCKED | 已分配 | 调度器已分配账号并推入 Redis，Worker 即将或正在领取 |
| Processing | 发送中 | Worker 正在用 Puppeteer/TextNow 执行发送 |
| Sent | 已发送 | 发送成功 |
| Failed | 失败 | 发送失败，可点「重试」 |
| Retry / Paused | 重试 / 已暂停 | 其他中间状态 |

**为什么一直显示「待处理」？**  
说明任务始终是 **Pending**，调度器没有把它变成 LOCKED/Processing。常见原因即上文：Worker 没起、Redis 不可用、没有 Ready 账号、账号没有配置代理、或任务被「时间熵」排到未来执行。按「二～四」检查并修复后，状态会依次变为 已分配 → 发送中 → 已发送（或 失败）。
