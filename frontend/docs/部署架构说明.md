# 部署架构说明：为什么和「本地电脑」有关？

## 一、谁在用「本地电脑」？

- **桌面应用（Tauri）**：安装并运行在**每个用户的电脑**上。用户开机打开应用，应用才会运行。
- **后端 API + 数据库**：部署在**服务器**上（例如 43.160.225.156 / hkd.llc），**一直开机**，对外提供接口。

也就是说：  
- 只有「桌面应用」会跑在用户自己的电脑上。  
- 「后端 + 数据库」只跑在机房/云上的服务器，**不依赖用户家里的电脑**。

---

## 二、桌面应用连的是哪里？

应用一启动，请求的是**服务器地址**，不是用户本机：

- 前端/桌面默认 API 地址：`https://hkd.llc/api`（见 `frontend/src/api.ts` 中 `VITE_API_BASE_URL`）。
- 所以：**用户本地电脑关机 = 只是这台电脑上的应用不能用**；服务器还在跑，其他用户照常使用。
- **只有服务器关机或故障时**，所有人都会连不上、应用才都不能用。

结论：**应用是否可用，取决于服务器是否在线，不取决于某台用户电脑是否关机。**

---

## 三、之前的「检查」为什么写的是本地？

- 脚本 `check_db_and_api.ts` 里之前用 `http://127.0.0.1:3000`，是给**运维/开发**用的两种场景：
  1. **在服务器上执行**：SSH 到服务器，在服务器上跑脚本 → 检查的是**服务器本机**的数据库和 API（此时「本地」= 服务器）。
  2. **在开发机上执行**：本机起后端、本机跑脚本 → 检查的是**你开发环境**的数据库和 API。

- 和「用户家里的电脑关不关机」**没有关系**：用户电脑上只装桌面客户端，不跑这个检查脚本，也不跑后端或数据库。

---

## 四、如何检查「线上」是否正常？

- 从**任意一台能上网的电脑**运行检查脚本，把 API 指到**生产环境**即可，例如：
  ```bash
  cd backend
  API_BASE=https://hkd.llc/api npx tsx scripts/check_db_and_api.ts
  ```
- 这样检查的是：**服务器上的 API 是否正常**（数据库检查仍依赖当前机器上的 `.env`；若在服务器上执行且 `.env` 是服务器的，则 DB 也是服务器的）。

总结：**用户本地电脑关机只会影响该用户自己；应用能否使用，取决于部署后端和数据库的那台服务器是否在线。**
